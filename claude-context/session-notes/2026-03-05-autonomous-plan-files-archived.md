## Key Decisions

### DEC-001: Semantic Naming Convention for Plan Archives

**Context**: Temporary plan files used random-word triplets (e.g., `mossy-kindling-quail.md`) generated by the planning system. These names are memorable during active work but meaningless for future reference. Need a naming convention that preserves searchability while maintaining uniqueness.

**Options Considered**:
1. **Topic-session format** (`{topic}-s{session}.md`)
   - ✅ Human-readable topic description
   - ✅ Session number preserves temporal context
   - ✅ Grep-friendly for both topic and timeframe
   - ✅ Unique (session numbers never repeat)
   - ❌ Requires manual name selection

2. **Keep random names, add metadata**
   - ✅ No renaming work required
   - ❌ Not grep-friendly
   - ❌ Meaningless filenames
   - ❌ Requires reading file content to understand topic

3. **UUID or timestamp-based names**
   - ✅ Guaranteed unique
   - ❌ No semantic meaning
   - ❌ Harder to navigate

**Decision**: Use `{topic-description}-s{session}.md` format. Topic descriptions come from plan file titles/summaries, session numbers come from plan metadata.

**Consequences**:
- Filenames are immediately meaningful and grep-friendly
- Future agents can search by topic (`grep -l "heartbeat" *.md`) or timeframe (`ls *-s7*.md`)
- Session numbers provide cross-reference to session notes in `claude-context/session-notes/`
- Manual naming required during archive (acceptable one-time cost)

### DEC-002: Categorised Index Structure

**Context**: With 22+ plan files in the archive, need a way to navigate the knowledge base without reading all files or relying on filesystem grep alone.

**Options Considered**:
1. **Categorised list with descriptions**
   - ✅ Semantic grouping aids discovery
   - ✅ One-line descriptions provide context
   - ✅ Markdown format allows linking
   - ❌ Manual categorisation effort

2. **Flat alphabetical list**
   - ✅ Simple to generate
   - ❌ No semantic structure
   - ❌ Harder to discover related plans

3. **Tag-based metadata in frontmatter**
   - ✅ Machine-parsable
   - ❌ Requires modifying plan files
   - ❌ Breaks original plan file preservation

**Decision**: Create categorised index with one-line descriptions. Six categories emerged naturally from the plan topics: Leo Heartbeat & Identity, Agent Architecture, Admin UI & Dispatch, Discord Integration, Infrastructure & Sync, Licences App.

**Consequences**:
- Index provides semantic entry points into the knowledge graph
- Categories group related architectural decisions
- One-line descriptions enable quick scanning without opening files
- Future additions require manual index updates (acceptable maintenance cost)
